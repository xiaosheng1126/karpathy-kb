# 个人开发者工具链

## 结论

个人开发者需要的不是单个“效率神器”，而是一条可以持续复用的工具链。

当前已经稳定适合放进这条链的方向有两类：

1. Google 工具组：负责 AI、知识、资料、实验、分析和增长反馈。
2. Cloudflare 工具组：负责前端托管、边缘 API、存储、隧道、监控和轻量基础设施。

这两类工具组合起来，能覆盖从“想法出现”到“上线反馈”的最小闭环。

## 适用场景

- 独立开发新工具、新站点或 AI 原型。
- 维护个人知识库、需求池和技术资料库。
- 快速把本地服务、调试页、演示页公开出去。
- 做轻量产品后端、文件存储、状态页、监控和通知桥接。
- 让 AI、文档、实验、发布和反馈各自有固定工具承接。

## 判断依据

**判断**：个人开发者的工具链应该按“工作流”组织，而不是按“软件列表”组织。
- 置信度：high
- 有效期：2026-12
- 来源：2026-06-24-google-tools-stack-for-indie-developers.md
- 不确定性：如果个人项目转向团队化，工具链会向协作系统和权限系统迁移。

**判断**：Google 工具组更适合承接思考、知识、实验和增长反馈。
- 置信度：high
- 有效期：2026-12
- 来源：2026-06-24-google-tools-stack-for-indie-developers.md
- 不确定性：部分能力会受地区和套餐影响。

**判断**：Cloudflare 工具组更适合承接发布、边缘计算、存储、隧道和监控。
- 置信度：high
- 有效期：2026-12
- 来源：2026-06-23-awesome-cloudflare-personal-developer-platform.md
- 不确定性：免费额度、平台限制和账号策略会变化。

**判断**：先把工具串成闭环，比继续扩充工具数量更重要。
- 置信度：high
- 有效期：2026-12
- 来源：2026-06-24-google-tools-stack-for-indie-developers.md
- 不确定性：闭环边界会随着项目类型变化。

## 方法或流程

### 当前主链路

```text
Gemini / AI Studio -> NotebookLM -> Colab -> Drive / Docs / Sheets -> Lighthouse / PageSpeed -> Analytics / Search Console
```

### 基础设施辅链

```text
Cloudflare Pages / Workers -> R2 / D1 / KV -> Tunnel -> Monitoring / Notifications
```

### 维护原则

- 新工具只在旧链路出现瓶颈时加入。
- 任何工具都必须回答：它替代了什么？提升了哪一步？
- 不把“知道有这个工具”误判成“已经建立了工作流”。

## 限制

- 这是一篇总入口，不是完整百科。
- 具体工具细节应该回到对应主题页，不要在这里堆砌参数和功能列表。
- 工具链会随着个人项目变化，需要按季度复盘。
